• 我去把 SelAcc@10% 的评估路径对齐到代码级别，给你一个确定答案。

• 不是“单个 12-tuple 里”的 SelAcc@10%。
  也不是“直接把一个 64-run 全排序后，按 top/mid/bottom 现算 band 分，再拿这个 band 分做 SelAcc@10%”。

  现在代码里的流程是：

  - 训练时
      - 对每个 12-tuple，先按模型打分排序。
      - 再按 4/4/4 分 band，用 R = alpha*x + beta*y + gamma*z 算 reward。
      - 这个 reward 只用于训练目标，用来学一个 best_model 的打分方向。
  - 验证 / 推理时
      - 对一个题的 64 个 run，不是只看一个 12-tuple。
      - 会随机采很多个 blind 12-tuple（当前默认 1024 个）。
      - 每个 tuple 里，best_model 给 tuple 内 12 个 run 各打一个分。
      - 然后把某个 run 在它出现过的所有 tuple 里的分数做平均，得到这个 run 的最终分数 score_best。
      - 所以这里确实已经变成“对该题全部 64 个 run 的进一步排序分数”。
  - SelAcc@10% 怎么算
      - 用的就是这些最终 per-run 分数，不是单 tuple reward。
      - 代码在 nad/ops/bestofn_extreme8.py::_score_metric 里：
          - 先把 validation 里所有 run 的最终分数和正确性收集起来；
          - 再按最终分数全局排序；
          - 取前 10%；
          - 计算其中正确样本的平均值。
      - 也就是说，summary 里的 SelAcc@10% 是基于最终聚合后的 run 排序。

  最关键的一句：

  - band 的 alpha/beta/gamma 只影响“怎么训练出打分器”
  - 最终报出来的 SelAcc@10% 看的是这个打分器对 run 的最终排序效果

  再补一个你可能关心的细节：

  - 对单个 dataset cache（一个 bundle）来说，SelAcc@10% 不是“每题各取 64 run 的前 10% 再平均”；
  - 它是把该 bundle 里所有验证题的 run 分数拼在一起后，再取全局前 10% 来算。
  - 最后 summary 里的 mean 又是对各 dataset bundle 做平均。
